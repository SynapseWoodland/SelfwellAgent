# 业务多轮对话追问 Skill

> **目的**：解决你这几次对话里反复出现的痛点——"用户在跑 Harness 期间反复追问业务问题，AI 不知道该答 / 答什么深度 / 用什么语气"。
> **触发场景**：详见 §一。
> **核心方法**：4 步结构化澄清（§三）+ 5 类追问路由（§四）+ 业务回答模板（§五）。

---

## 一、触发条件（何时读本 skill）

| 触发信号 | 例子 | 本 skill 的响应 |
|---------|------|----------------|
| 用户在 Harness phase 中**追问业务** | "这个 FR 我该怎么拆 / 客户问 X 我怎么回" | 进入 4 步澄清法 |
| 用户说"业务上 XX 我理解对吗" | "客户画像应该是 25-35 还是 30-45？" | 进模板 §五.A |
| 用户对 ORCHESTRATOR synthesis 有质疑 | "第三步为什么建议用 Redis" | 进模板 §五.B |
| 用户在 SIGN_OFF 反复问"要不要发布" | "今天能上吗 / 影响如何 / 风险" | 进模板 §五.C |
| 用户说"我看不懂 ATDD 那条 / TDS 这句话" | "TDS 字段枚举值为什么是 X" | 进模板 §五.D |
| 用户对话 3 轮以上追问同一主题 | "再想想 / 还有什么 / 还有吗" | 进入 §六"对追问的反追问" |

**反触发**（不要进入本 skill）：
- 用户问工程规范 / 真源 / 流水线逻辑 → 进 `pr-gate` / `coding-standards` / `ad-tdd`
- 用户问 PR 失败原因 → 进 `pr-gate` workflow 日志
- 用户问"我应该修什么文件" → 进对应 phase 的 reviewer / executor

## 三、4 步结构化澄清法（核心方法）

> 每个业务追问，**都必须走完这 4 步**再回答。不要一次性给答案——这是解决"AI 答非所问"的根本。

### Step 1：你真的在问什么？（Purpose 目的）

把用户的模糊问题**翻译成"决策请求 / 知识请求 / 反馈请求 / 情绪请求"四类之一**：

| 类别 | 用户表面语言 | 真实问题 | 输出物 |
|------|------------|---------|--------|
| **决策请求** | "我该怎么拆 / 选哪个" | 想要 AI 帮决策 | 给 2-3 个选项 + 推荐 |
| **知识请求** | "为什么 XX / 什么原理" | 想要 AI 解释 | 给清晰解释 + 引用真源 |
| **反馈请求** | "我这样写对吗 / 行不行" | 想要 AI 验证 | 给 PASS/FAIL + 修复路径 |
| **情绪请求** | "我焦虑 / 我担心 / 我不想做" | 想要 AI 共情 | 给共情 + 减缓方案 + 决策选项 |

**反模式**（要警惕）：把"决策请求"当"知识请求"回答（"为什么不用 Redis" 答错了——用户要的是决定）。

### Step 2：边界（Constraints 约束）

追问**任何会改变范围的前提**：

- 是否有真实用户场景？
- 时间窗口？预算？人力？
- 法规/合规约束？（V3 §5.1 合规四层）
- 是否依赖其他 FR？

**示例**：

```
用户："PRD 里 FR-PLAN-04 客户回滚要几秒？"
Step 2 应反问：
- "您指的是'从发起撤回请求到撤回完成'的总耗时，还是'撤回按钮点击到 UI 反馈'的感知耗时？"
- "退回后计划已经部分触达的服务（如已推送通知）怎么办？"
```

### Step 3：影响（Impact 影响）

列出**回答问题的连锁后果**：

- 此回答会让谁做什么？
- 是否会触发某 phase 重跑？
- 是否有 ADR / TDS 改动？
- 对在跑的其他 FR 的影响？

**示例**：

```
"如果您把 FR-PLAN-04 拆成 3 个 FR-PRA-04-A / B / C，会触发：
- workflow.yaml 从 PRA-04 一次走完变成 3 次 harness run
- 每 PR 都需 pr-gate 6 项 PR-Gate，CI 资源 × 3
- 集成测试回归计划 × 3
- SIGN_OFF 阶段工作量上升"
```

### Step 4：决策（Decision 选择）

把上述三步综合，给**选项 + 推荐 + 折中**：

```
A. 拆为 3 FR（推荐）：隔离性好，能独立回滚 ← 适合您担心的"误操作"
B. 合并为 1 FR（不推荐）：工作量小，但没法部分发布
C. 折中：拆为 2 FR（A/B 合并为 FRA-04-FULL，C 单独）—— 推荐如果拆分 + 集成工作量太大

我的推荐：A，理由是您之前的 5 次追问集中在"误操作害怕"，A 给您最大保护。
```

## 五、业务回答模板（5 套）

### A. 追问 PRD 业务定义

**结构**：

```
1. 我理解的您的问题
2. V3 真源（标注 §X.Y + ADR-XXXX if exists）
3. 同行业 / 同类项目通常做法（参考来源）
4. 我推荐拆法（2-3 选 1）
5. 拆完后会发生什么（adb 列出各项后果）
6. 下一步建议
```

**示例**：

> 您问："PRD 里 FR-PLAN-04 客户撤回计划到底要几秒感知？"
>
> 1. 您关心的是 UI 按钮点击到用户感知"撤回成功"的延迟（perceived latency），而不是系统响应时间（system latency）
> 2. V3 §10.2 NFR 对"撤回"未明文规定，只规定"推送 P95 ≤ 1s"
> 3. 行业参考：飞书/钉钉撤回的感知 ≤ 2s，微信朋友圈撤回 ≤ 3s
> 4. 我推荐：
>    - 感知 ≤ 2s（推荐）：给用户"瞬时撤回"错觉
>    - 感知 ≤ 5s（保守）：匹配服务端 P95
>    - 不规定：让前端写明"撤回中…"
> 5. 副作用：≤ 2s 需要前端乐观更新（不等服务端），后端做异步补偿
> 6. 下一步：要不要我现在起草 TDS 把这条锁定？

### B. 追问方案选型

**结构**：

```
1. 您在质疑的技术选型
2. 选 / 不选的 3 条理由（每条 ≥ 1 行真实依据）
3. 真实成本（不仅是钱：人力 / 时间 / 维护 / 培训）
4. ADR 引用 + 影响范围
5. 折中方案
```

**示例**：

> 您问："TDS 里为啥用 Postgres 不用 MySQL？"
>
> 1. V3 §1 已切换 MySQL → PG
> 2. 切换原因（按 ADR-0002）：
>    - Vector 内置（pgvector）vs MySQL + ES 双栈（→ 1 套而非 2 套）
>    - JSONB 原生支持 vs MySQL 用 JSON（PG 索引更好）
>    - 区间类型（daterange）vs MySQL B-Tree（按时间窗查询）
> 3. 真实成本：PG 10 人天培训 + pgvector 4 人天；MySQL 双栈 5 人天 + ES 集群 8 人天
> 4. 影响的 ADR：0002 / 0005 / 0010 / 0011
> 5. 折中：先用 MySQL 上线，PG 仅用于向量；W4 后再迁移

### C. 追问上线节奏

**结构**：

```
1. 当前 phase + 已签状态
2. 上线 checklist（每项完成度 %）
3. 风险评估（人 / 系统 / 数据 / 业务 / 合规 五维）
4. 推荐发布时间 + 灰度节奏
5. rollback 策略
```

### D. 追问验收标准

**结构**：

```
1. ATDD 当前 Scenario 覆盖范围（正常 / 边界 / 异常 三态）
2. 隐含的"没写出的"边界条件
3. 用户旅程中断点（4 个常见点）
4. 推荐补充的 Scenario
```

### E. 追问运营数据

**结构**：

```
1. 数据回流层级（前端埋点 → 后端事件 → 用户旅程 → 业务指标）
2. 当前 OA 看板 / Grafana dashboard 链接（如有）
3. 1 个能立即跑的验证 SQL
4. 推荐监控的 5 个核心指标
```

## 六、对追问的反追问（防止循环追问）

当用户对话已经追问同一主题 3 轮以上时，**主动停下**并说：

> "我注意到您已经追问我 X 三次了。我担心我始终没答到您真正的问题。
> 让我用 4 步法重问：
>
> 1. 您是真的在问 [决策 / 知识 / 反馈 / 情绪]？
> 2. 是不是我漏了什么前提？
> 3. 这个回答会影响 [哪条 FR / 哪个 phase / 哪个用户]？
> 4. 比起之前几次，您现在更想听 [A / B / C] 中的哪个？"

**目的**：终止循环。**这是 anti-loop 设计**——避免 AI 越答越偏。

## 七、与 INTERRUPT_REVIEW phase 的衔接

V2 phase 16 `INTERRUPT_REVIEW` **专用于本 skill 触发的追问**：

- 用户追问 → AI 读本 skill → 给出答案 → **必须**写入 `state.interrupt_log`
- INTERRUPT_REVIEW phase 在 SIGN_OFF 之前必跑：review**所有** interrupt 是否处理完
- 若未关闭：在 PR 描述列"未关闭追问清单"

### interrupt_log schema

```yaml
interrupt_log:
  - timestamp: "2026-07-18T15:30:00+08:00"
    phase: 04  # ATDD
    question: "客户撤回感知的延迟到底是多少"
    answer_given: "1. 感知延迟 vs 系统延迟 2. 飞书 ≤ 2s 3. 推荐 ≤ 2s"
    closed: false
    resolved_in: null  # 或 PR 编号 / TDS 章节
```

`interrupt_budget` 默认每 run 5 次追问打断，超出升级主会话用户。
