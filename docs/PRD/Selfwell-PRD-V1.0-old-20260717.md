## 0. 产品愿景

### 0.1 一句话定义

**Selfwell** 是一款 AI 驱动的自律成长陪伴产品,通过多模态 AI 诊断 + 21 天方案 + 每日打卡闭环 + 主动回忆技术,帮助用户在**不评判、不承诺效果、不量化指标**的温柔陪伴中实现自律成长。

### 0.2 目标用户

| 用户分层 | 描述 | 核心诉求 |
|---------|------|---------|
| **Primary** | 18-35 岁女性,久坐/体态/皮肤焦虑 | 每天 10-20 分钟跟练,21 天养成自律习惯 |
| **Secondary** | 希望建立长期健康习惯,但自律动力不足 | 通过每日打卡+AI 记忆陪伴,降低放弃率 |
| **Tertiary** | 需要"不焦虑"的自我观察工具 | 不追求"变瘦/变美",而是"与自己和解" |

**非目标用户**:
- 需要医疗治疗的用户(产品明确拒绝)
- 追求快速见效的用户(产品无效果承诺)

### 0.3 核心问题陈述

| 维度 | 现状痛点 | Selfwell 如何解决 |
|------|---------|----------------|
| **自律工具** | 打卡应用强迫坚持、失败=否定用户 | 温柔陪伴,失败不被评判 |
| **诊断服务** | 医美/护肤建议充满承诺和焦虑 | "养护方向"而非"治疗",无数字指标 |
| **AI 助手** | AI 评判用户"比之前好/差了" | Recall Safety 三层防线,绝对不评判 |
| **方案匹配** | 方案固定,无法个性化 | AI 多模态诊断+全网视频匹配,每日动态方案 |
| **遗忘曲线** | 21 天后忘记起点,动力消失 | Day 7/14/21 主动回忆,让用户看见自己的变化 |

### 0.4 核心价值主张

```
"和自己走过的每一天,都是进步。"
```

**Selfwell 不承诺:**
- ❌ "坚持 21 天皮肤会变好"
- ❌ "每天打卡 BMI 会下降"
- ❌ "效果比同龄人好"

**Selfwell 只承诺:**
- ✅ 每天一个专属视频方案
- ✅ Day 7/14/21 温柔回顾你的成长
- ✅ 你的数据只有你自己看得到

| 用户类型 | `lifecycle_stage` | 语义定义 | 进入条件（任一） | 退出条件 |
|---------|-------------------|---------|----------------|---------|
| **新用户** | `new` | 第一次启动、刚完成 M1-M6 引导 + 诊断 + 方案生成的用户。处于"建立认知 + 形成基础方案"阶段 | (a) 首次启动创建 user record；(b) `days_since_signup == 0 && M6 未完成` | M6 完成 → `maturing` |
| **过渡期用户** | `maturing` | 已完成 21 天方案生成但**尚未进入稳定期**的用户（30 天观察期）。仍在用新用户稿，但可看到部分老用户专属节点预览（M10 Day7 抱抱卡） | `M6 完成` | `days_since_signup >= 7 && completed_tasks_d1_to_d7 >= 14` → `matured` 跃迁评估通过；否则停留在 `maturing` |
| **老用户** | `matured` | 已度过 7 天观察期、进入"稳定陪伴 + 抱抱卡颁发"阶段的用户。走老用户稿，享 M10/M11/M12 抱抱卡 + M15 智能管家 | (a) `maturing → matured` 跃迁评估通过；(b) v1 一次性脚本回填 | 失联 ≥ 30 天 → `retained` |
| **复发用户** | `retained` | 失联 30 天后回访的老用户。走老用户稿 + M23 复发纪念弹窗 | `now - last_active_at >= 30 days` | 回访当天完成任意 1 个任务 → `matured` 回迁 |

##### 判定入口字段（4 个核心维度）

| 字段 | 类型 | 说明 | 数据来源 |
|------|------|------|---------|
| `users.signup_at` | TIMESTAMP | 账号创建时间 | 注册时一次性写入 |
| `users.lifecycle_stage` | ENUM | 当前所处阶段，4 选 1 | §0.x.7.3 切换规则每次启动重算 |
| `users.last_active_at` | TIMESTAMP | 最近一次成功打卡 / 浏览 / 互动时间 | 任何 M 节点 commit 时更新 |
| `task_completion` 聚合 | TABLE | `tasks.user_id, day_index(1-21), completed_at` 每日打卡记录 | M7 打卡节点写入 |

##### 派生指标（在 backend 计算，不存库）

```
days_since_signup        = current_date - signup_at::date
completed_tasks_d1_to_d7 = COUNT(task_completion WHERE day_index BETWEEN 1 AND 7)
total_completed_tasks    = COUNT(task_completion WHERE completed_at IS NOT NULL)
days_since_last_active   = current_date - last_active_at::date
```

##### 关键边界说明

1. **"新用户"是过程状态，不是永久标签**：用户不会永远是"新用户"。新用户稿是**入门期的视觉稿**，21 天方案生成（M6）完成即失去"新用户"语义。
2. **"老用户"不是"用得久的用户"的同义词**：单纯 `days_since_signup` 大不等于老用户。**必须同时满足** `days_since_signup >= 7` **且** `completed_tasks_d1_to_d7 >= 14`。
3. **`retained` 是过渡态**：复发用户首次回访当天完成 1 个任务即回迁 `matured`，**不退化为 `maturing`**。
4. **v1 老用户回填**：v2 启动时一次性脚本把所有 v1 老用户 `lifecycle_stage = 'matured'` + `days_since_signup = now - signup_at`，避免他们走新用户稿。
5. **`maturing` 观察期不弹"抱抱卡预告"**：M10 Day7 抱抱卡在 `maturing` 阶段**不出现**（PRD §1.8 抱抱卡禁止表达），只有进入 `matured` 后才进入 Day 7/14/21 颁发序列。

#### 0.x.7.3 切换判定规则（双用户旅程的核心边界）

> **2026-07-15 新增**：本节给出"何时走新用户稿 / 何时走老用户稿 / 何时弹复发纪念"的**强制判定规则**，作为后端 API、前端路由、DB 字段设计的唯一真源。

**核心字段**：`users.lifecycle_stage ∈ {new, maturing, matured, retained}`

**判定顺序**（按优先级由高到低，首条命中即终止）：

| 优先级 | 触发条件 | `lifecycle_stage` | 走哪套稿 | 备注 |
|--------|---------|-------------------|---------|------|
| 1 | 首次启动（无 user record） | `new` | **新用户稿** M1-M6 全套 | 强制走完 M6 方案生成 |
| 2 | `users.lifecycle_stage == 'new' && M6 完成` | `maturing` | **新用户稿** M7-M14 | 30 天观察期，仍走新稿 M9 等新用户专属节点 |
| 3 | `days_since_signup >= 7 && completed_tasks_d1_to_d7 >= 14` | `maturing → matured` 跃迁评估 | **老用户稿** M7/M8/M13/M14 + M10 Day7 抱抱卡 | 满足 d1-d7 完成 14/21 任务 |
| 4 | `days_since_signup >= 21 && total_completed_tasks >= 60` | `matured` | **老用户稿** 完整 | M11/M12 抱抱卡 |
| 5 | 失联 ≥ 30 天回访 | `matured → retained` | **老用户稿** + **复发纪念弹窗**（视觉节点 8b） | 弹窗基于 M9 心情日记历史选段 |
| 6 | `retained` 用户首次完成当天任务 | `retained → matured` 回迁 | **老用户稿** | 不退化为新稿 |

> **实现约束**：
> 1. **判定在后端**：`GET /api/v1/journey/next` 一次性返回 `lifecycle_stage` + `next_node_id`（M 系列 ID）+ `journey_spec_id`（`new_journey` / `old_journey` / `retained_journey`）。
> 2. **前端不再做分支判断**：拿到 `journey_spec_id` 后**直接整稿加载**，不混搭两稿节点。
> 3. **复发纪念弹窗优先级**：弹窗节点 8b 必须在 M1 之后、M15（P03a Tab）之前插入。
> 4. **数据回填**：v1 老用户一次性脚本回填 `lifecycle_stage = 'matured'` + `days_since_signup = now - signup_at`，避免回访时走新稿。

### 1.1 AI 多模态诊断(场景:诊断三步卡)

**S03 对应:附录 A1**

#### 问题

用户在开始自律旅程前,需要了解自己的现状("我哪里最需要关注")。

#### 核心交互

```
[用户] → [上传 3 张照片] → [AI 多模态诊断] → [获得个性化报告] → [选择是否开始方案]
```

**上传 3 张照片(业务规则 F4-b + F4b1 已锁):**
- 用户上传 1~3 张人体照片(MVP 上限 3 张)
- 系统 AI 自动给每张照片识别 1 个身体部位(6 部位 + unclassified)
  - 6 部位:`face`(脸) / `head`(头) / `shoulder_neck`(肩颈) / `waist`(腰) / `leg`(腿) / `overall_look`(整体气色)
  - `unclassified`:不分部位,仅 mood_photo 允许
- 每张照片下方显示 AI 识别的部位标签 + **[修改]**按钮
- 用户可点击修改,从 6 部位下拉选择 1 个覆盖 AI 识别结果
- **允许部位重复**(例如 3 张都是 face)
- 提交后,若 distinct 部位数 < 3,报告页**软提示**"可补部位以获得更完整方案"(不阻断)
- 至少 1 张照片的部位 ≠ `unclassified` 才能提交(避免零信息)

#### AI 诊断输出(来自 `backend/app/llm/schemas.py` V5.2.1)

```json
{
  "directions": [
    {
      "title": "肩颈放松",
      "description": "每天 5 分钟肩颈拉伸...",
      "video_id": "uuid-xxx",
      "level": "轻度 | 中度 | 重度"
    }
  ],
  "tags": ["久坐", "肩颈", "面部干燥", "眼疲劳"...],
  "summary": "整体状态良好,主要关注肩颈放松..."
}
```

#### 诊断报告页

- 显示 3-5 条改善方向(含视频推荐)
- 显示 7-14 个诊断标签
- 显示 AI 整体总结(50-100 字)
- 每个方向显示严重度等级(`轻度/中度/重度`)
  - **轻度**:建议动作,自主选择
  - **中度**:推荐视频方案,提醒咨询专业医师
  - **重度**:提示"建议您咨询专业医师",不承诺效果

#### 业务规则

| 规则 | 内容 |
|------|------|
| **照片数** | 1-3 张,上限 3 |
| **部位识别** | AI 自动多标签分类,用户可改 |
| **重复部位** | 允许,distinct 计数 |
| **置信度不足** | 软提示,不阻断 |
| **重度判定** | 提示转医师,AI 不提供医疗建议 |
| **诊断成本** | ≤ ¥0.15/次 |
| **诊断耗时 P95** | ≤ 20 秒(异步 SSE 流式) |
| **报告缓存** | 7 天复用(期间重新诊断返回缓存) |

#### 边缘情况

| 情况 | 处理 |
|------|------|
| 上传失败 | 重试提示,最多 3 次,超时提示换网络 |
| AI 超时 | 降级到规则引擎,返回基础建议 |
| 用户中途退出 | 不保存,重新上传 |
| 照片不合格(非人体) | 微信 sec 校验拦截 + 提示"请上传包含人体部位的照片" |
| 用户不想做诊断 | 跳过,直接进入首页(但首页提示"先做诊断获得个性化方案") |

### 1.2 21 天方案生成与视频匹配(场景:方案生成)

**S04 对应:附录 A4**

#### 问题

用户做完诊断后,需要一个可执行的、个性化的每日方案。

#### 核心交互

```
[诊断完成] → [方案生成中] → [21 天每日任务列表] → [每日推送视频任务]
```

#### 方案结构

```
plan = {
  id: UUID,
  user_id: UUID,
  days: [
    { day: 1, video_id: UUID, phase: 1 },
    { day: 2, video_id: UUID, phase: 1 },
    ...
    { day: 8, video_id: UUID, phase: 2 },
    ...
    { day: 15, video_id: UUID, phase: 3 },
    ...
    { day: 21, video_id: UUID, phase: 3 }
  ],
  status: "active" | "completed" | "abandoned",
  started_at: DATE,
  completed_at: DATE
}
```

**21 天分三阶段:**
| 阶段 | 天数 | 视频特点 |
|------|------|---------|
| Phase 1 | Day 1-7 | 入门级,5-10 分钟,轻柔难度 |
| Phase 2 | Day 8-14 | 进阶,10-15 分钟,适中难度 |
| Phase 3 | Day 15-21 | 巩固,15-20 分钟,进阶难度 |

#### 视频匹配算法

```
score(video) = 0.5 × 标签匹配度 + 0.3 × 时长适配度 + 0.2 × 难度适配度

标签匹配度   = Jaccard(视频标签集, 诊断标签集)
时长适配度   = 1 - |视频时长 - 用户偏好时长| / 60
难度适配度   = 1 - |视频难度 - 用户强度等级| / 5
```

**视频池要求:**
- 启动期 ≥ 50 条视频
- 成熟期 ≥ 200 条视频
- 跳过率 > 30% 触发告警

#### 业务规则

| 规则 | 内容 |
|------|------|
| **方案触发** | 诊断完成后自动生成,或用户主动进入"生成方案"页 |
| **视频匹配** | 按诊断标签 + 用户档案(focus_parts / intensity)匹配 |
| **每天 1 个视频** | 不可跳过,不可选 |
| **已完成方案** | 不可修改,标记 completed |
| **放弃方案** | 标记 abandoned,可新建方案(新方案从 Day 1 开始) |
| **多方案** | 最多同时 1 个 active 方案;历史方案存档 |
| **方案生成耗时 P95** | < 2 秒 |

#### 边缘情况

| 情况 | 处理 |
|------|------|
| 视频池不足 50 条 | 提示"方案匹配中,请稍后重试";不阻断,降级到随机推荐 |
| 21 天内提前完成(连续打卡提前到达 Day 21) | 方案提前 completed;抱抱卡 Day 7/14/21 仍按实际完成天数触发 |
| 用户中途不想做了 | 标记 abandoned;首页引导重新生成方案 |
| 方案生成失败 | 降级到默认方案(Phase 1 全套入门视频) |

#### 打卡完成话术(来自 `docs/data/ack-pool.yaml` 30 条)

**触发条件**:打卡完成(feeling 可选填,≤ 50 字)

| 话术示例 | 出现场景 |
|---------|---------|
| "今天又迈出了一步,真的很棒。" | 任意 |
| "自律的路上,你并不孤单。" | 任意 |
| "完成比完美更重要,你做到了。" | 任意 |

**ACK 话术禁用规则(来自 `docs/data/ack-pool.yaml`):**
- 绝对不量化("坚持了 7 天了,真棒" ❌)
- 绝对不对比("比之前好多了" ❌)
- 绝对不承诺("继续坚持皮肤会变好" ❌)
- 字数上限:30 字

#### 连续打卡中断规则

| 连续未打卡天数 | 推送话术 |
|--------------|---------|
| 1 天(19:00 提醒) | "今天累了就休息,明天的你还在这里。" |
| 2 天(10:00 提醒) | "想休息一天也可以,先把今天的你抱抱。我们慢慢来。" |
| 3 天+ | Persona slight_hug 态,轻量关心,不强推送 |

#### streak 天数计算规则(已锁)

| 场景 | streak 值 | 说明 |
|------|-----------|------|
| Day 1 打卡 | streak = 1 | 从第 1 个打卡日起算 |
| 连续打卡 | streak 累加 | 每天 +1 |
| 前一天没打卡(中断) | streak 重置为 1 | 中断后重新从 1 开始 |
| 补卡 | streak 不补 | 补卡只更新状态,不补 streak |

> **streak 计算起点**:从用户第 1 个打卡日起算,与 plan.start_date 无关,前一天未打卡即中断,streak 归 1。

#### 补卡规则(已锁)

|| 场景 | streak 值 | 说明 |
|------|-------|-----------|------|
| 补 Day 1 卡 | streak 不补 | 补卡只更新打卡状态(is_supplemental=true),不补 streak |
| 补卡后当天打卡 | streak 正常累加 | 补卡完成后,当天打卡 streak 继续从补卡 streak 值起算 |
| 补卡后次日打卡 | streak 正常累加 | 补卡后连续打卡,streak 连续 |

> **补卡语义**:补卡是「补记录」,不是「补坚持」。产品不评判用户「是否真的做了」,只记录用户「想要记录」。这与「不评判」价值观一致。

#### plan.current_day 与 streak_days 两套独立逻辑(已锁)

| 字段 | 计算规则 | 场景示例 |
|------|---------|---------|
| **plan.current_day** | 按方案自然日推进,Day 1 没打卡,Day 2 打卡,current_day = 2 | 方案到期判断(第 21 天完成) |
| **streak_days** | 从第 1 个打卡日起算,前一天没打卡归 1,Day 1 没打卡,Day 2 打卡,streak = 1 | 抱抱卡触发/打卡连续天数展示 |

> 两套逻辑互不影响:方案进度自然走,打卡连续天数独立计算。

#### 打卡进度展示双层规则(已锁)

|| 展示位置 | 字段 | 计算规则 | 文案示例 |
|------|---------|------|---------|---------|
| **进度环** | 今天 Tab 环 | `plan.current_day` / 21 | 按方案自然日推进 | "3 / 21" |
| **副文案** | 今天 Tab 环下方 | `streak_days` | 从第 1 个打卡日起算,前一天未打卡即重置为 1 | "已坚持 3 天 · 慢慢来" |
| **概览卡** | 21 天方案入口卡 | `streak_days` | 同上 | "21 天方案 · 你走过 3 天" |

> **语义区分**:进度环告知「方案走到哪了」,副文案强调「你真正坚持了多久」—— 前者是事实,后者是鼓励。

#### 业务规则

| 规则 | 内容 |
|------|------|
| **打卡步骤** | ≤ 2 步(看视频 + 点按钮) |
| **感想字数** | ≤ 50 字(可选填) |
| **打卡时间** | 任意,按完成时间记 |
| **重复打卡** | 唯一约束 (user_id, plan_id, day) 不允许重复 |
| **打卡反馈** | 30 条 ACK 随机,不出现 LLM 生成话术 |
| **Persona 切换** | 按打卡状态 + 用户情绪触发 persona_state 切换 |
| **streak 计算** | 从第 1 个打卡日起算,前一天未打卡即重置为 1 |

#### 边缘情况

| 情况 | 处理 |
|------|------|
| 视频播放失败 | 降级:标清 → 截图 → 文案兜底 |
| 用户跳过视频直接打卡 | 允许,但打卡按钮点击前须等待 ≥ 30 秒 |
| 用户已打卡但想补充感想 | 30 分钟内可编辑 feeling,超出不提供编辑 |

### 1.4 智能管家主页(场景:智能管家)

**S02 对应:附录 A2**

#### 问题

用户需要一个随时可交互的 AI 陪伴者,在不同场景(诊断/打卡/反馈/记忆/闲聊)下提供个性化的、不评判的陪伴。

#### 核心交互

```
[用户打开 App] → [进入智能管家主页(P03a)] → [看到入口卡] → [与 AI 对话]
```

#### P03a 主页入口卡(4 张)

| 入口卡 | 触发条件 | 内容 |
|--------|---------|------|
| **智能诊断** | 用户未完成诊断或超过 30 天未诊断 | "上传照片,了解现在的自己" |
| **心情日记** | 用户进入 M7 任意路径 | "记录今天的心情,哪怕只有一句话" |
| **主动回忆** | Day 7/14/21 触发 + 用户主动入口 | "和你走过的 7 天,想看看吗?" |
| **直接输入** | 任意时刻 | AI 对话输入框 |

#### Persona 状态机(来自 ADR-0015)

| 状态 | 触发条件 | 话术示例 |
|------|---------|---------|
| `warm`(温暖) | 默认 | "慢慢来,{nickname}。" |
| `neutral`(中性) | 闲聊/不知道说什么 | "嗯嗯,我在听。" |
| `slight_hug`(轻度拥抱) | 用户表达疲惫/失败/连续未打卡 | "想休息一下也没关系。" |

#### Persona 状态切换规则

- **warm → slight_hug**:用户连续 7 天无 feedback 或连续 2 天未打卡
- **warm → medical_guarded**:用户输入含医疗/症状词(痤疮/皮肤病/药物/医生等)
- **slight_hug → warm**:用户完成打卡或提交 feedback
- **medical_guarded → warm**:用户换话题

#### AI 回复规则

| 规则 | 内容 |
|------|------|
| **回复耗时 P95** | ≤ 3 秒 |
| **Persona 话术** | 4 态话术来自 Persona 合同,不出现 LLM 自由生成 |
| **合规红线** | Recall Safety 100+ 词三层防线;Persona 永不评判用户变化 |
| **未知意图** | 走 SmartRouter unknown 路径(落库到 intent_unknown_logs),返回"嗯嗯,这个我还不太懂,换个话题吧" |

#### 业务规则

| 规则 | 内容 |
|------|------|
| **会话超时** | 30 分钟无交互,会话自动关闭 |
| **同时 open 会话** | 同一用户同时最多 1 个 open session |
| **意图分类** | 8 类:module_redirect / read_query / recall / recall_ack / feedback_ack / feedback_create / medical_reject / unknown |
| **会话历史** | 30 分钟内可回溯;超出按 new session 处理 |

#### 基线问候规则模板池(不走 LLM,T-D-4 已锁)

| 档位 | 触发条件 | 模板文案示例 |
|------|---------|------------|
| (a) 未开始 | 用户未做诊断 | "早上好,我是 Selfwell。今天想做什么?" |
| (b) 已诊断+今日打卡完成 | 已诊断 + 今日已完成打卡 | "今天练完了,{nickname} 真的可以。我们明天见。" |
| (c) 今日未打卡 | 今日未打卡 | "今天累了就休息,明天的你还在这里。" |
| (d) 连续 ≥2 天未打卡 | 前一天+更早均未打卡 | "想休息一天也可以,先把今天的你抱抱。我们慢慢来。" |

> 模板来源:待建 `docs/data/greeting-pool.yaml`,由 S02 场景废弃的 LLM Prompt 改写。


### 1.6 主动回忆 Day 7/14/21(场景:主动回忆)

**S07 对应:附录 A7**

#### 问题

21 天自律旅程中,用户在第 7/14/21 天需要一个温柔的"回头看",帮助他们看见自己的变化,而不是焦虑于"还剩多少天"。

#### 核心交互

```
[Day N 早上 8:00] → [系统自动加载反馈素材] → [AI 生成回忆卡] → [推送通知]
         ↓
    [用户打开 App] → [主动回忆页面] → [看到温柔的回顾] → [获得鼓励语]
```

#### Day N 触发规则(D1 已锁:仅对 active plan 触发)

**触发锚点**:active plan.started_at 起算
- Day 7:plan.started_at + 7 天,08:00 触发
- Day 14:plan.started_at + 14 天,08:00 触发
- Day 21:plan.started_at + 21 天,08:00 触发

> **started_at 语义**:用户在 M4 交付页点击「从今天开始」按钮时写入,强调「你真正开始行动的那天」,不是「方案生成的那天」。若用户跳过 M4 直接进入 App,started_at = now()。

**触发条件(双层)**:
1. **日期条件**:plans.status = 'active' + deleted_at IS NULL + today >= plan.started_at + N 天
2. **打卡条件**:用户**连续打卡满 N 天**(从第 1 个打卡日起算,前一天没打卡即中断归 1)
   - 若打卡中断但日历日期到达:Recall 气泡**不自动推送**
   - 但用户主动点「问问过去的自己」**不受此限制**,实时生成

**未打开 App 的处理**:
- 当日 10:00 再推送 1 次提醒
- 若用户仍未打开:不重复推送,当天 20:00 静默;回忆卡在 App 内"待看"状态保留 7 天

#### 回忆素材来源

| 素材 | 来源 | 说明 |
|------|------|------|
| 文字心情 | `feedback.feedback_type='mood_text'` | 用户写的感想 |
| 周期照片 | `feedback.feedback_type IN ('period_photo', 'plan_compare_photo')` | 带部位的照片 |
| 打卡记录 | `checkins` 表 | 已完成的视频任务 |
| 非评判摘要 | AI 从以上素材生成 | 总结"这 N 天你关注了什么,感觉如何",不提"改善/变化" |

#### Recall Safety 三层防线(来自 ADR-0017)

| 层 | 防线内容 | 说明 |
|---|---------|------|
| Layer 1 | Prompt 硬编码 | AI 绝对不评判用户前后变化/颜值/数字 |
| Layer 2 | 100+ 敏感词库 | 4 组:前后评判(50+词) / 效果承诺(22+词) / 数字评判(15+词) / 外貌评判(20+词) |
| Layer 3 | 安全兜底 | 命中敏感词时,替换为 5 条 safe_fallback 随机 |

**敏感词库真源**:`docs/data/recall-forbidden-words.yaml`

**绝对禁止的词(示例):**
```
进步/变好/瘦了/胖了/颜值/BIM/体重/三围/变漂亮了/比我好...
```

**安全兜底示例:**
- "自律是一辈子的事情,不在于某一天的完美。"
- "每个人的节奏不一样,不必和别人比较。"

#### 业务规则

| 规则 | 内容 |
|------|------|
| **触发锚点** | active plan.created_at 起算 |
| **触发时间** | 08:00(仅 active plan) |
| **ai_summary** | ≤ 200 字,不提变化/数字 |
| **ai_encourage** | ≤ 80 字,温柔鼓励 |
| **Safety 通过** | 100+ 敏感词 3 层全过,方能推送给用户 |
| **素材缺失** | 不足 3 条 feedback 时,补充系统打卡摘要填充 |
| **无素材** | 无任何 feedback + checkin → 显示默认回忆卡(不暴露零数据尴尬) |

#### 边缘情况

| 情况 | 处理 |
|------|------|
| Day 7 触发但用户第 3 天才建 plan | 实际建 plan 后 + 7 天才触发(不是注册时间) |
| Day 14 触发但用户提前完成方案(连续打卡到 Day 14) | 按实际完成天数触发,不提前 |
| 用户主动进入 Day N 回忆入口 | 不受 08:00 限制,实时生成,同 Safety 检查 |
| 用户注销后恢复 | 恢复 active plan,继续累计 |

#### 自定义时间跨度（用户主动"问问过去的自己"）

> **2026-07-15 v1.0 新增**：用户主动点「问问过去的自己」时，UI 给出**默认快捷选项 7 天 / 14 天 / 21 天** + **自定义 chip 入口**。本节规定自定义跨度的边界规则。

**前端入口**：`00-olduser-journey-v2.html` 节点 9 回忆页快捷 chip 数组：`switchAskPast(3|7|14|21|...)`，3 与自定义边界由本节 CSR-1 ~ CSR-8 约束。

> **v2.1 修订注**：设计稿节点 9 快捷 chip 实际包含 **3 / 7 / 14 / 21 + 自定义**（不只是 7/14/21）。3 天 chip 是面向"刚完成 Day 3 的新用户"的近跨度入口；自定义 chip 承担 1 ~ 99 天任意跨度。

| 规则编号 | 规则 | 边界值 | 后端含义 |
|---------|------|--------|---------|
| CSR-1 | 时间跨度下界 | `>= 1` 天 | `recall_span_days >= 1` |
| CSR-2 | 时间跨度上界 | `<= plan.current_day` | 若 `plan.current_day=10`，用户最多问"10 天前"，**问 20 天前直接拒绝** |
| CSR-3 | 计划不存在 | — | `plans.status != 'active'` → 返回 E_RECALL_INVALID_SPAN(13xxx)，UI 提示"还没有开始的日子可以回忆" |
| CSR-4 | 反馈数据不足 | — | 该跨度区间内 `feedback < 3 条` → 触发默认摘要兜底（同 §1.6 "无素材" 行为） |
| CSR-5 | AI 输出长度 | 同 §1.6 ai_summary ≤ 200 字 | Recall Safety 三层防线 100% 走全 |
| CSR-6 | 触发频率限流 | 单用户 ≤ 5 次/24 小时 | `recall_sessions.created_at` 24h 滑动窗口 |
| CSR-7 | 时区 | 按用户所在时区计算"今天 0 时" | 后端用 `users.timezone`（如缺失则 `Asia/Shanghai`） |
| CSR-8 | 数据采样 | 区间内随机 ≤ 20 条 feedback | 防 prompt 过长 |

**API 契约**（同步 SPEC-M12 落地）：

```
GET /api/v1/recall?span_days=7
GET /api/v1/recall?span_days=14
GET /api/v1/recall?span_days=21
GET /api/v1/recall?custom_span_days=10    # CSR-2 ≤ plan.current_day

Response 200: {
  "ai_summary": "...",   # ≤ 200 字
  "ai_encourage": "...", # ≤ 80 字
  "span_days": 10,
  "plan_current_day": 10,
  "feedback_count": 7
}

Response 422: { "code": "E_RECALL_INVALID_SPAN", "message": "..." }
```

### 1.8 抱抱卡/分享海报(场景:抱抱卡)

#### 核心交互

```
[Day 7/14/21 打卡完成] → [生成抱抱卡海报] → [用户选择分享/保存]
```

#### 抱抱卡文案规则(来自 SPEC-M10 禁止词)

**允许的表达:**
- "和你走过的 {n} 天"
- "和你一起 · 第 {n} 天"
- "我们一起走到了这里"
- "我们走到了中点"（Day 14 专用）

**绝对禁止的表达:**
- ❌ "我坚持了 7 天"
- ❌ "已坚持 21 天"
- ❌ "效果比之前好多了"
- ❌ "皮肤变好了"
- ❌ "已坚持 {streak_days} 天"（抱抱卡不用 streak_days,用方案进度）

> **抱抱卡语义**:里程碑式庆祝 —— 「我们一起走到了这里」,强调「陪伴」而非「坚持」。文案基于方案进度(plan.current_day),而非打卡连续天数(streak_days)。

#### 抱抱卡内容

| 内容 | 来源 |
|------|------|
| 分享文案 | AI 生成(Recall Safety 检查后),基于 user.nickname + day 数 |
| 配图 | 用户 `feedback.feedback_type='plan_compare_photo'` 照片(优先);无则用 mood_photo |
| 视频截图 | 当天打卡视频封面 |
| 二维码 | 小程序码(带用户参数) |

#### 业务规则

| 规则 | 内容 |
|------|------|
| **触发点** | Day 7 / Day 14 / Day 21 **打卡完成后** |
| **触发条件** | plan.current_day 达到 7/14/21 **且** 当天打卡完成;用户有 `plan_compare_photo` 照片;无则降级到普通打卡完成卡 |
| **海报尺寸** | 适配朋友圈 3:4 比例 |
| **有效期** | 永久有效(不像 export URL 有 TTL) |
| **分享方式** | 微信小程序分享(原生能力) |

> **触发语义**:抱抱卡是里程碑庆祝,「我们一起走到了这里」。方案进度满 N 天(plan.current_day)即触发,不要求连续打卡。打卡中断用户依然可获得抱抱卡,只是文案会温柔一些。

## 3. 合规与隐私

> **叙述轴**:什么人 / 什么问题 / 什么价值
> **合规目标**:让用户感受到"我的数据是安全的",而不是"我们被监管了"

### 3.1 合规四层纵深(产品规则层,不提 RLS/PG 实现细节)

| 层 | 名称 | 产品表现 |
|---|------|---------|
| L1 | 输入拦截 | 微信 sec 校验 + 关键词拦截(医疗/功效承诺词) |
| L2 | LLM 约束 | System Prompt 硬编码合规红线 + JSON Schema 约束输出 |
| L3 | 输出网关 | 逐 chunk 关键词检查 + 采样 LLM 分类 |
| L4 | 异步复审 | 社区内容三段式审核(关键词/采样 LLM/人工) + safety_audit_logs 记录 |

### 3.2 合规红线(产品面向用户的话术规则)

| 红线 | 禁止 | 替代 |
|------|------|------|
| 1 | 医疗表述(治疗/治愈/医生/处方/病症) | "养护方向" / "生活方式建议" |
| 2 | 效果承诺(会变白/会变小/会提升) | 不承诺,只说"建议" |
| 3 | 前后对比判断(比之前好/进步了) | "你的感受很重要" |
| 4 | 数字量化(BMI/体重/三围) | 不量化 |
| 5 | 颜值/外貌评判(变漂亮/颜值高) | 不评判 |
| 6 | 强推打卡("必须坚持") | "今天做了就很棒" |

### 3.3 隐私保护原则

| 原则 | 内容 |
|------|------|
| **逐支用户隐私隔离** | 用户上传的照片只有本人可见 |
| **数据最小化** | 诊断原图 7 天后自动清除(保留 AI 诊断结论,不保留原图) |
| **透明告知** | 隐私协议首次登录强制展示 |
| **用户控制权** | 可导出数据(export_jobs)、可注销账户(15 天冷静期) |
| **无第三方分享** | 用户数据不卖给/分享给任何第三方 |

### 3.4 安全网关(产品承诺层)

| 承诺 | 说明 |
|------|------|
| **上传安全** | 微信 sec 校验 + 内容安全 API |
| **传输安全** | HTTPS 加密传输 |
| **存储安全** | MinIO/COS 私有桶,签名 URL 访问 |
| **访问控制** | JWT Token + UnionID 鉴权 |

### 3.5 心理健康危机识别与升级

> **背景**:Selfwell 目标用户(18-35 岁女性,体像焦虑)是高心理健康风险群体,产品必须具备危机识别与专业资源转介能力,这既是伦理责任,也是 PIPL/网络安全法的潜在要求。

#### 3.5.1 危机分级

| 级别 | 触发条件 | 产品响应 |
|------|------|---------|---------|
| **L-Crisis** (立即危机) | 用户表达自杀意念/自伤行为("活着没意义"、"想死"、"割腕"等) | 停止一切 AI 交互,立即展示危机资源卡 |
| **L-Serious** (严重情绪) | 用户表达持续抑郁/无助/绝望情绪,且无明确触发事件 | AI 暂停提供建议,展示温柔倾听+专业资源 |
| **L-Medical** (医疗急迫) | 用户表达急性心理症状(惊恐发作、解离等) | 建议立即就医,提供急诊指引 |

#### 3.5.2 L-Crisis 升级流程(立即执行)

```
[用户输入触发 crisis 关键词]
    ↓
[AI 立即停止生成 → 展示危机响应卡]
    ↓
┌─────────────────────────────────────────┐
│  🌿 我在这里陪着你                           │
│                                          │
│  如果你现在感到很难熬,                        │
│  请不要一个人扛着。                          │
│                                          │
│  📞 全国心理援助热线: 400-161-9995           │
│     (24 小时免费)                          │
│                                          │
│  📞 北京心理危机研究与干预中心: 010-82951332  │
│     (24 小时)                             │
│                                          │
│  🚨 如有紧急危险,请立即拨打 120 或 110        │
│                                          │
│  [我需要帮助]  [我只是想聊聊]               │
└─────────────────────────────────────────┘
    ↓
["我只是聊聊" → 恢复轻度 AI 陪伴,但继续监控输入]
["我需要帮助" → 展示完整资源列表,不恢复 AI 对话]
```

#### 3.5.3 L-Crisis 禁用回复(绝对禁止)

以下回复模式在 crisis 场景下**绝对禁止**:

- ❌ "我无法回答医疗问题,建议您咨询专业医师"(冷漠、不即时)
- ❌ "建议您拨打心理援助热线"(单一资源、无情感连接)
- ❌ 任何 AI 生成的建议、分析、或共情长文(可能加重危机)
- ❌ 任何形式的打卡提醒或方案推送

#### 3.5.4 危机词表管理

危机词表(`docs/data/mental-health-crisis-words.yaml`)独立于 `medical-reject-words.yaml` 管理:

| 词表 | 用途 | 严重级别 |
|------|------|---------|
| `mental-health-crisis-words.yaml` | 自杀/自伤/危机表达 | **L-Crisis** (最高) |
| `medical-reject-words.yaml` §dangerous_behavior | 重度心理/精神症状 | L-Serious |
| `medical-reject-words.yaml` §medication_treatment | 医疗/用药咨询 | L-Medical → 医疗转介 |

#### 3.5.5 数据记录

危机事件按以下规则处理:

| 规则 | 内容 |
|------|------|-----|
| **不记录对话内容** | 用户在危机状态下的输入不存储于 AI 对话历史 |
| **匿名统计** | 仅记录"某日触发 L-Crisis N 次",用于产品安全监测 |
| **审计日志** | `safety_audit_logs` 表记录 crisis 触发事件(不含对话内容),保留 90 天 |

#### 3.5.6 心理健康合规对齐

| 监管要求 | 对齐措施 |
|---------|---------|
| PIPL Art.13 — 同意 | 用户协议中明确告知心理健康支持功能的存在与限制 |
| PIPL Art.14 — 敏感信息 | 危机词表数据不关联用户身份标识 |
| 《互联网信息服务深度合成管理规定》 | AI 对话中明确"我不是专业心理咨询师"的身份声明 |
| 行业伦理 | 产品团队应定期审查危机词表,建议每季度一次 |
