# Harness Engineering 全量评估 Prompt

> 本 Prompt 用于指导子 Agent 对 SelfwellAgent 项目的 Harness Engineering 进行全面评估。
> 评估者应从第三方视角，结合业界最佳实践，以批判性思维对当前方案进行审视。

---

## 二、评估维度

### 2.1 架构评估

**评估要点**:
1. **模块解耦程度**: Harness 各组件（Dispatcher/Executor/Orchestrator/Reviewer）是否职责清晰、低耦合？
2. **可扩展性**: 新增 Agent 类型 / 新增工作流阶段是否需要修改核心代码？
3. **状态管理**: 状态机是否覆盖了所有业务场景？状态转换是否有完善的错误处理？
4. **容错机制**: 单点故障如何处理？Agent 执行失败后的恢复策略？
5. **并发模型**: 多个 Agent 同时执行时的协调机制？

**业界参考**:
- AutoGen / LangGraph 的状态机设计
- CrewAI 的角色隔离机制
- Microsoft AutoDev 的流水线架构

### 2.2 流程评估

**评估要点**:
1. **端到端流程完整性**: 从需求输入到代码提交的全链路是否覆盖？
2. **阶段划分合理性**: 分析 → 设计 → 开发 → 测试 → 部署的阶段划分是否清晰？
3. **阶段间衔接**: 上一阶段的输出如何传递给下一阶段？
4. **回退机制**: 某阶段失败后如何回退或重试？
5. **人工介入点**: 哪些节点需要人工审批？介入时机是否合理？

**核心场景验证**:
> 场景: 用户说"继续完成剩余所有步骤"，此时已有 PRD 和 SRS 文档。
> 期望: Agent 应自动生成后续文档（SDS/TDS）、执行 ATDD 开发、运行测试、提交 commit。
> **评估**: 当前流程是否能支持这一场景？有哪些断点？

### 2.3 Agent 角色分配评估

**评估要点**:
1. **角色定义清晰度**: 每个 Agent 的职责边界是否明确？
2. **角色协作机制**: Agent 间如何通信？消息格式是否统一？
3. **角色优先级**: 冲突时哪个 Agent 有最终决策权？
4. **角色扩展性**: 新增角色是否需要修改多个文件？
5. **角色能力边界**: 每个角色能做什么 / 不能做什么？

**当前角色列表**:
- Backend Agent
- Frontend Agent
- QA Agent
- DevOps Agent
- Security Agent
- Harness Orchestrator
- Harness Dispatcher

### 2.4 中断与人机交互评估

**评估要点**:
1. **中断触发条件**: 哪些情况会触发人工中断？是否覆盖所有必要场景？
2. **中断时机合理性**: 中断是否在正确的决策点？过早或过晚？
3. **恢复流程**: 人工处理后如何恢复自动流程？
4. **交互接口**: 人工如何输入反馈？输入格式是否有约束？
5. **超时处理**: 人工长时间未响应如何处理？

**必须中断的场景**（检查是否覆盖）:
- [ ] 需求歧义 / 理解不一致
- [ ] 技术选型争议
- [ ] 安全 / 合规风险
- [ ] 预算超限（LLM 月预算 ¥700）
- [ ] 测试失败且 Agent 无法自动修复
- [ ] Prompt 改动导致 Golden Set 回归失败
- [ ] PR 需要人工 Code Review

### 2.5 文档质量评估

**评估要点**:
1. **文档完整性**: 每个阶段的文档模板是否包含所有必要章节？
2. **文档一致性**: 同一条目在不同文档中的描述是否一致？
3. **文档可执行性**: 文档中的指令是否能被 Agent 直接执行？
4. **文档可验证性**: 文档中的验收标准是否可量化、可测试？
5. **文档时效性**: 如何保证文档与代码同步更新？

**文档列表**（需逐一检查）:

| 文档类型 | 模板文件 | 检查项 |
|----------|----------|--------|
| SRS | `docs/requirements/SELFWELL-MVP-SRS.md` | 功能需求、非功能需求 |
| ATDD | `harness/atdd/` | 验收测试场景 |
| SDS | 待评估 | 系统设计、接口定义 |
| TDS | `docs/spec/TDS-*.md` (待检查) | 技术规格、测试用例 |
| SPEC | `docs/spec/SPEC-M*.md` | 模块规格说明 |
| PRD | `docs/PRD/Selfwell-PRD-V1.1.md` | 业务目标、用户故事、KPI |

### 2.6 文档依赖关系评估

**评估要点**:
1. **依赖链路**: PRD → SRS → SDS → TDS → ATDD → Code → Test 的依赖是否清晰？
2. **引用一致性**: 下游文档引用上游内容时，是否验证引用的有效性？
3. **版本对齐**: 多文档协同修改时，版本号是否同步？
4. **冲突检测**: 文档间描述冲突时如何检测和解决？
5. **孤儿文档**: 是否有文档未被任何流程引用？

**依赖图检查**:
```
PRD ──┬──→ SRS ──┬──→ SDS ──┬──→ TDS ──┬──→ Code
      │          │          │          │
      └──────────┴──────────┴──────────┴──→ ATDD ──→ Test
```

### 2.7 文档与代码一致性评估

**评估要点**:
1. **描述准确性**: 文档描述的功能是否与实际代码行为一致？
2. **接口一致性**: 文档中的 API 接口是否与实际实现一致？
3. **路径正确性**: 文档中引用的文件路径是否真实存在？
4. **版本匹配**: 文档描述的是哪个版本的代码？
5. **过时检测**: 如何发现过时的文档内容？

### 2.8 反馈机制评估

**评估要点**:
1. **反馈来源**: 反馈来自哪些渠道（测试 / Review / 用户 / CI）？
2. **反馈分类**: 如何区分不同类型的反馈（Bug / 优化 / 安全）？
3. **反馈路由**: 不同类型的反馈是否路由到正确的 Agent？
4. **反馈闭环**: 反馈处理后是否通知到发起方？
5. **反馈积累**: 反馈经验是否沉淀为 Pattern / Lesson？

**反馈链路检查**:

```
测试失败 ──────→ 开发 Agent ──────→ 修复代码
                   ↑
                   │
人工 Review ───────┘

Golden Set 回归失败 ──→ Prompt 调优 ──→ 重新评估

Rule 拦截 ─────────→ 人工澄清 ──→ 规则修正
```

### 2.9 调度合理性评估

**评估要点**:
1. **调度触发**: 什么条件触发调度？是否有死锁风险？
2. **调度策略**: 串行 / 并行 / 优先级调度如何选择？
3. **负载均衡**: 多任务时的资源分配策略？
4. **调度日志**: 调度决策是否可追溯？
5. **调度性能**: 调度本身的延迟是否可接受？

## 三、评估输出格式

### 3.1 问题分类

| 严重程度 | 定义 | 数量 |
|----------|------|------|
| 🔴 Critical | 导致流程无法继续 | - |
| 🟠 High | 严重影响效率或质量 | - |
| 🟡 Medium | 中等影响，可优化 | - |
| 🟢 Low | 小问题，建议改进 | - |

```markdown
## 问题 #N: [问题简述]

- **严重程度**: 🔴 Critical / 🟠 High / 🟡 Medium / 🟢 Low

- **影响分析**: [对业务/开发/质量的影响]
- **根因分析**: [为什么会发生这个问题]
- **建议修复**: [具体的修复方案]
- **优先级**: P0 / P1 / P2 / P3
```

### 4.1 业界最佳实践参考

在评估时应参考以下资源：

1. **Microsoft AutoDev** (https://github.com/microsoft/autogendev)
   - 自动化开发流水线
   - AI Agent 协作机制

2. **LangGraph** (https://github.com/langchain-ai/langgraph)
   - 状态机设计
   - Agent 编排

   - 多 Agent 角色定义
   - 任务协作

4. **AutoGen** (https://github.com/microsoft/autogen)
   - Agent 间对话
   - 人机协作

5. **Harness.io** (https://harness.io/)
   - CI/CD 流水线最佳实践
   - 持续验证

6. **DORA Metrics**
   - 部署频率
   - 变更前置时间
   - 变更失败率
   - 恢复时间

### 4.2 评估步骤

1. **现状调研**: 读取所有 Harness 相关文件
2. **流程追踪**: 模拟执行关键场景（如"继续完成剩余步骤"）
3. **问题发现**: 按评估维度逐一检查
4. **根因分析**: 追溯问题根源
5. **方案设计**: 提出具体修复建议
6. **优先级排序**: 按影响程度排序

## 六、示例问题

以下是一些示例问题，用于启发评估思路（实际评估时应发现更多）:

### 示例 1: 阶段间衔接问题
> **场景**: 用户已有 PRD 和 SRS，要求继续完成所有步骤。
> **期望**: 自动生成 SDS → TDS → ATDD → 执行开发 → 测试 → Commit
> **问题**: 当前 Dispatcher 是否知道 SDS/TDS/ATDD 的生成顺序和触发条件？

### 示例 2: 中断时机问题
> **场景**: 开发 Agent 生成了一段违反 coding-standards 的代码。
> **期望**: Rule 在 L0 检查时拦截，而非等到 CI 失败。
> **问题**: 当前是否有 Local Rule 检查机制？

### 示例 3: 反馈闭环问题
> **场景**: QA Agent 发现了一个安全漏洞。
> **期望**: 漏洞信息反馈给开发 Agent，修复后重新测试。
> **问题**: 当前是否有漏洞 → 修复的反馈路由？

### 示例 4: 文档可执行性问题
> **场景**: ATDD 文档描述了验收测试场景。
> **期望**: Agent 可直接将 ATDD 转换为可执行的 pytest 测试用例。
> **问题**: ATDD 模板是否有结构化字段支持自动化转换？

## 七、注意事项

1. **批判性思维**: 不要假设当前设计是对的，要质疑每一个决策
2. **用户视角**: 站在最终用户（开发者/产品经理）的角度评估体验
3. **可操作性**: 建议必须具体可执行，避免空洞的"建议改进"
4. **优先级排序**: 资源有限时，先修复影响最大的问题
5. **持续改进**: 评估不是终点，应建立机制支持持续优化
